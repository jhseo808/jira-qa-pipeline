## 4. Phase 1-B — 서비스 분석 답변

> **TC 작성 전 이 섹션을 반드시 채운다. 답변이 TC 설계의 근거가 된다.**

**Q1. 이 서비스에서 사용자가 가장 믿어야 하는 핵심 데이터는?**

{예: 상품 판매가 / 차트 순위 / 잔여 재고 수 / 집계 수치}
→ 이 데이터가 틀리면 사용자에게 직접적 피해가 발생한다.

**Q2. 사용자가 수행하는 핵심 액션 3가지는?**

1. {예: 상품 검색 → 상세 이동}
2. {예: 장바구니 담기}
3. {예: 필터/정렬 변경}
→ 각 액션이 TC가 된다. 흐름 중 끊기는 지점은 Robust TC가 된다.

**Q3. 사용자에게 "상태"를 어떻게 알려주는가?**

{예: 품절/재고 있음 배지, 신규/인기 태그, 처리중/완료 상태 표시}
→ 상태 표시가 API 필드값과 일치하는지 검증한다.

**Q4. 이 서비스의 데이터는 언제 기준인가? 사용자가 알 수 있는가?**

{예: 실시간 순위, 오늘 기준 가격, 마지막 업데이트: N분 전}
→ 날짜·갱신 기준이 화면에 표시되는지 확인한다.

**Q5. 금전적·법적·안전 책임이 있는 데이터는?**

{예: 최종 결제가, 이자율, 약관, 처방 정보 / 없으면 "해당 없음"}
→ 계산 정확성·소수점·반올림까지 검증한다.

**Q6. 외부 의존성(API, 캐시, CDN) 실패 지점은?**

{예: 상품 이미지 CDN, 결제 API, 검색 인덱스, 실시간 데이터 스트림}
→ 각 의존성 실패 시 Fallback TC를 만든다.

**절차**

1. API 응답에서 상태 필드({예: isNew, isOutOfStock, rankChange})값을 확인한다.
2. DOM에서 해당 항목의 배지/태그 요소를 확인한다.
3. 날짜·갱신 기준 텍스트를 추출하고 형식을 확인한다.
4. 한글·특수문자 인코딩 깨짐 여부를 확인한다.

**기대 결과**

- 상태 필드값과 화면 배지가 **일치**한다 (예: `isNew: true` → "NEW" 배지 노출).
- 날짜 기준 텍스트가 존재하고 유효한 날짜 형식이다.
- 텍스트에 `\uFFFD` (인코딩 깨짐)가 **없다**.
- HTML 엔티티(`&amp;`, `&#`)가 미변환 상태로 **노출되지 않는다**.

**자동화 매핑**

| 항목 | 값 |
|------|-----|
| 스펙 파일 | `tests/{기능}-ui.spec.ts` |
| assert 예시 | `expect(text).not.toContain('\uFFFD'); expect(dateText).toMatch(/\d{4}/)` |
| 비고 | 상태 필드명은 실제 API 명세 확인 후 수정 |
