# 테스트케이스 명세

## 1. 문서 목적과 읽는 사람

이 문서는 **QA 요청 티켓 `{JIRA_QA_TICKET}`**에 대해, 무엇을 어떤 조건으로 검증할지 **재현 가능한 단위(테스트케이스)**로 정의한다.  
설계·개발·QA·자동화 담당이 **같은 기준**으로 리뷰하고, Step 3 이후 **Playwright 스펙·POM**과 **1:1 또는 N:1로 대응**시키기 위한 기준이 된다.

---

## 2. 참조 및 전제

| 항목 | 경로 또는 설명 |
|------|----------------|
| QA 계획서 | `qa_plan_ref`에 명시된 파일 (보통 `output/{ticket}/qa_plan.md`) |
| 요구사항·스펙 | 계획서 §3 Test Basis·Jira·API 명세에 따름 |
| 테스트 환경 | 계획서 §9 및 본 문서 §6 |

**전제**: 본 TC는 계획서의 **범위(In Scope)·Exit Criteria·리스크**와 모순되지 않게 작성한다.

## 3. 용어

| 용어 | 의미 |
|------|------|
| **TC** | Test Case. 아래 §8 형식의 최소 검증 단위. |
| **DATA-xxx** | 사전 정의한 **테스트 데이터 세트** 식별자. §7에서 정의한다. |
| **유형 태그** | TC의 성격. 예: `functional`, `regression`, `smoke`, `robust`, `nfr`. 한 TC에 복수 가능. |
| **Robust** | 동일 기능에 대해 **환경·데이터·실패 조건** 변화(경계, 부정, 지연, 빈 응답 등)를 검증하는 TC. |
| **NFR** | Non-functional. 성능·가용성·보안·접근성 등 **기능 외 품질 요구**를 검증하는 TC. |
| **추적 ID** | 계획서의 리스크(R1…), 요구사항 ID, NFR 항목 ID 등 **출처 표기**. |

## 4. 테스트 데이터 카탈로그 (필수 권장)

동일한 “아이디어”의 TC라도 **입력이 다르면 별도 TC**로 두거나, 한 TC 안에서 **DATA-xxx를 명시**한다.  
자동화 시에는 이 ID가 `fixtures`·`test-data.ts`와 연결된다.

| 데이터 ID | 설명 | 예시 값 / 조건 | 비고 |
|-----------|------|----------------|------|
| DATA-001 | 정상 기본값 | {예: 로그인된 일반 사용자} | |
| DATA-002 | 경계값 | {예: 순위 1위·100위} | |
| DATA-003 | 부정·예외 | {예: API 500 응답 시뮬레이션} | 수동/모킹 여부 |
| DATA-… | | | |

민감 정보(실계정·토큰)는 **값 자체를 문서에 쓰지 말고**, “스테이징 계정 A”, “환경 변수 `E2E_TOKEN`”처럼 **참조만** 기술한다.

## 7. TC 작성 규칙

1. **TC ID**는 티켓 내에서 중복되지 않게 부여한다 (예: `TC-001` …).
2. **전제 조건**이 없으면 “없음”으로 명시한다.
3. **기대 결과**는 측정 가능하게 쓴다. NFR은 **수치·임계값**을 넣는다 (예: 응답 시간 &lt; 1초).
4. **Robust**: 정상 케이스와 **조건이 다르면** 별도 TC로 분리하는 것을 권장한다.
5. **자동화 예정**이면 “자동화 매핑”에 파일·스펙 이름을 Step 3 완료 후 채운다.
