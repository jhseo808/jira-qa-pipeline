# Step 2: 테스트케이스 작성 (testcases)

[← 공통](common.md) · [인덱스](README.md)

---

## 1. 이 단계가 하는 일

Step 2는 **QA 계획서를 기반으로 사용자가 서비스를 신뢰할 수 있는 수준의 테스트케이스**를 생성한다.

**핵심 원칙**: TC는 "기능이 동작하는가"가 아니라 **"사용자가 이 서비스를 믿을 수 있는가"** 를 기준으로 설계한다.

사용자는 다음 조건이 충족될 때 서비스를 신뢰한다.

| 신뢰 조건 | 깨지는 순간 |
|----------|------------|
| 보이는 데이터가 진짜다 | 가격이 결제 시 달라진다 |
| 빠짐없이 보여준다 | 100개라더니 98개다 |
| 계산이 정확하다 | 할인율 적용이 틀렸다 |
| 상태를 명확히 알려준다 | 품절인지 모르고 담았다 |
| 어디서나 같은 결과다 | 모바일에서 가격이 다르다 |
| 오류 시 길을 잃지 않는다 | 흰 화면만 나온다 |
| 기다리지 않아도 된다 | 5초 이상 로딩이다 |
| 누구나 사용할 수 있다 | 키보드로 접근 불가다 |

## 3. TC 생성 절차

> **Claude Code는 Phase 0 → Phase 1 → Phase 2 → Phase 3 순서를 반드시 따른다.**
> **Phase 0 없이 TC를 작성하지 않는다. 문서만 보고 추측으로 작성하면 안 된다.**

---

### Phase 0 — 실제 페이지 분석 (playwright-cli 필수)

> **TC의 선택자·필드명·assert값은 모두 여기서 관찰한 실제 값을 사용한다.**
> 이 단계를 통과하지 않으면 TC는 추측 기반이 되어 Step 3에서 구현이 어렵거나 잘못된다.

#### 0-1. playwright-cli로 페이지 접속

`state["target_url"]`(또는 QA 계획서의 URL)로 playwright-cli를 사용해 페이지에 접속한다.

```
# 실행 예시 (playwright-cli 스킬 사용)
/playwright <target_url>
```

#### 0-2. 네트워크 요청 수집

페이지 로드 중 발생하는 XHR/Fetch 요청을 모두 기록한다.

| 수집 항목 | 목적 |
|----------|------|
| API endpoint URL 목록 | TC의 `page.route` 모킹 대상 확정 |
| 주요 응답 JSON 필드명 | 축 1 진실성 TC의 실제 필드명 확정 |
| 상태 필드 (isNew, status 등) | 축 4 신뢰 신호 TC의 필드명 확정 |
| 날짜·가격·계산값 필드 | 축 3 정확성 TC의 계산식 확정 |

#### 0-3. DOM 구조 확인

페이지에서 다음을 직접 확인하고 **실제 선택자**를 기록한다.

| 확인 항목 | TC에서 사용 목적 |
|----------|----------------|
| 항목 행 선택자 (반복 요소) | 축 2 완전성 — 항목 수 count |
| 핵심 텍스트 선택자 (제목, 가격, 순위 등) | 축 1 진실성 — API값과 DOM값 비교 |
| 이미지 선택자 | 축 6 복원력 — naturalWidth 체크 |
| 링크 선택자 | 축 4 신뢰 신호 — href 유효성 |
| 상태 배지 선택자 (NEW, 품절, HOT 등) | 축 4 신뢰 신호 — 배지 표시 일치 |
| 날짜/갱신 기준 텍스트 선택자 | 축 4 신뢰 신호 — 날짜 형식 검증 |
| 인터랙션 요소 (버튼, 탭, 필터, 폼) | 축 1·2 — 핵심 액션 TC |

#### 0-4. 신뢰 신호 육안 확인

| 확인 항목 | 있음/없음 | 내용 |
|----------|----------|------|
| 날짜/시간 표시 | | |
| 가격 구조 (원가→할인가→최종가) | | |
| 항목 수 표시 | | |
| 상태 배지 종류 | | |
| 인코딩 깨짐 여부 | | |
| Broken Image 존재 여부 | | |

#### 0-5. 관찰 결과를 test_cases.md에 기록

Phase 0에서 수집한 내용을 `test_cases.md`의 **"Phase 0 관찰 결과"** 섹션에 기록한다.
이 섹션은 TC 본문에 등장하는 모든 선택자·필드명의 **출처 근거**가 된다.

---

### Phase 1 — 변경사항 추출 + 서비스 분석 (TC 작성 전 필수)

> **Phase 1은 두 파트로 구성된다.**
> Part A는 이번 릴리즈에서 "무엇이 바뀌었는가"를 추출한다.
> Part B는 서비스 전반의 신뢰 구조를 분석한다.
> **둘 다 완료한 후 Phase 2로 진행한다.**

---

#### Part A — 변경사항 추출 (Change Impact Analysis)

QA 계획서와 Jira 티켓에서 **이번 릴리즈의 변경사항**을 전부 추출한다.
각 변경사항에 대해 아래 표를 채운다.

| 변경 ID | 변경 영역 | 이전(Before) | 이후(After) | 영향 범위 | TC 필요 여부 |
|---------|----------|-------------|------------|----------|-------------|
| C-001 | {예: DB 구조} | {예: 집계 테이블} | {예: 실시간 로그} | {예: API 응답 전체} | 필요 |
| C-002 | {예: 순위 로직} | {예: 단순 합산} | {예: 가중치 점수} | {예: rank, score 필드} | 필요 |
| C-003 | {예: Redis TTL} | {예: 10분} | {예: 3분} | {예: 캐시 갱신 주기} | 필요 |

**변경사항 추출 규칙:**
- "기능 추가", "로직 수정", "설정 변경", "의존성 업그레이드" 등 모든 유형을 포함한다
- 계획서에 명시된 변경이 없으면 "변경사항 없음 (순수 버그픽스)" 이라고 기재한다
- 변경 영역이 넓을수록 영향 범위가 크고, TC 수도 늘어난다

각 변경사항은 **두 종류의 TC**를 요구한다:

| TC 종류 | 목적 | 예시 |
|---------|------|------|
| **[변경 검증]** | 의도된 변경이 실제로 적용되었는가 | TTL이 정말 3분이 되었는가 |
| **[변경 회귀]** | 변경으로 인해 기존 기능이 깨지지 않았는가 | TTL 변경 후 캐시 만료 시 빈 화면이 없는가 |

---

#### Part B — 서비스 신뢰 분석 질문

QA 계획서를 읽고 아래 **6가지 분석 질문**에 답한다.
**답변 없이 TC를 작성하지 않는다.** 답변이 TC의 설계 근거가 된다.

---

**Q1. 이 서비스에서 사용자가 가장 중요하게 믿어야 하는 데이터는 무엇인가?**

> 예시: 가격, 순위, 재고 수량, 평점, 뉴스 날짜, 집계 수치, 처방 용량
> → 이 데이터가 틀리면 사용자에게 직접적 피해가 발생한다.
> → **Critical TC**의 기반이 된다. API 응답값과 UI 표시값을 실제 값으로 비교한다.

**Q2. 사용자가 이 서비스에서 수행하는 핵심 액션 3가지는 무엇인가?**

> 예시: "상품 검색 → 상세 페이지 이동 → 장바구니 담기"
> 예시: "차트 진입 → 곡 선택 → 재생"
> 예시: "대시보드 진입 → 날짜 필터 → 데이터 확인"
> → 각 액션이 TC가 된다. 액션 흐름 중 끊기는 지점을 Robust TC로 추가한다.

**Q3. 이 서비스에서 "상태"를 사용자에게 어떻게 알려주는가?**

> 예시: 품절/재고 있음, 신규/인기, 처리중/완료/실패, 온라인/오프라인
> → 상태 표시가 API 데이터와 일치하는지 검증하는 TC.
> → 상태가 변경될 때 UI가 즉시 반영되는지도 확인한다.

**Q4. 이 서비스의 데이터는 언제 기준인가? 사용자가 그것을 알 수 있는가?**

> 예시: "실시간 순위", "오늘 기준 가격", "최근 업데이트: 2분 전"
> → 날짜·시간·갱신 기준 표시가 화면에 있는지 확인하는 TC.
> → 없으면 신뢰 신호 부재로 버그 등록 기준이 된다.

**Q5. 이 서비스에서 금전적·법적·안전 책임이 있는 데이터는 무엇인가?**

> 예시: 가격, 이자율, 약관, 처방 정보, 납기일, 계약 조건
> → 이 데이터는 단순 표시 확인이 아니라 **계산 정확성·소수점·반올림**까지 검증한다.
> → 없으면 "해당 없음"으로 기재한다.

**Q6. 이 서비스가 외부 의존성(API, 캐시, DB, CDN)에 의해 실패할 수 있는 지점은 어디인가?**

> 예시: 외부 결제 API, 이미지 CDN, 검색 인덱스, 실시간 데이터 스트림
> → 각 의존성 실패 시 사용자에게 어떤 화면이 보여야 하는지를 TC로 만든다.
> → "흰 화면"이나 "JS 에러"가 아닌 명확한 Fallback이 있어야 한다.

### Phase 2 — TC 도출: 변경 검증 + 8대 신뢰 축

TC는 **두 그룹**으로 구성된다. 두 그룹 모두 작성한다.

---

#### 그룹 A — 변경 검증 TC (Change Verification)

> Phase 1 Part A에서 추출한 **변경사항 하나당 최소 2개** TC를 만든다.
> - `[변경 검증]`: 의도된 변경이 실제로 반영되었는가
> - `[변경 회귀]`: 변경이 기존 기능을 깨뜨리지 않았는가

**변경 검증 TC 작성 기준:**

변경 내용을 Before → After로 명확히 명시하고, After 상태를 직접 측정·비교한다.

```
TC 제목: [변경 검증] {변경 영역} — {이전값} → {이후값}
절차:
  1. 변경 후 동작을 직접 측정한다 (API 호출, DOM 확인, 타이밍 측정 등)
기대 결과:
  - After 상태가 명세대로 적용되었음을 수치로 확인한다
  - 예: "Redis TTL이 3분(180s)임을 Cache-Control 헤더에서 확인한다"
  - 예: "score 필드가 스트리밍+좋아요+공유 가중치 합산값임을 fixture와 비교 확인한다"
```

**변경 회귀 TC 작성 기준:**

변경의 영향 범위를 기준으로, 인접 기능이 여전히 정상 동작하는지 확인한다.

```
TC 제목: [변경 회귀] {변경 영역} 변경 후 {인접 기능} 정상 동작
절차:
  1. 변경된 영역에 의존하는 기존 기능을 실행한다
기대 결과:
  - 기존 동작이 변경 전과 동일하게 유지된다
  - 예: "TTL 3분 변경 후, 캐시 만료 시에도 빈 화면 없이 정상 데이터 표시"
  - 예: "순위 로직 변경 후, 100개 항목 누락·중복 없이 표시"
```

---

#### 그룹 B — 8대 신뢰 축별 TC

Phase 1 Part B의 답변을 바탕으로, 아래 **8개 축 각각에서 TC를 도출**한다.
각 축마다 최소 1개 TC를 만들거나, "N/A: {이유}"를 명시한다.

---

#### 축 1 — 진실성 (Truth): 보이는 데이터가 실제 데이터와 같은가

**핵심 질문**: API/DB의 값과 화면에 표시된 값이 정확히 일치하는가?

TC 도출 기준:
- Q1에서 파악한 핵심 데이터를 API 응답값과 DOM 값으로 직접 비교한다
- "JSON 응답이 존재한다", "길이 > 0" 수준의 assert는 이 축을 충족하지 않는다
- 반드시 **실제 값 비교** (`toBe(expectedValue)`)여야 한다

기대 결과 작성 기준:
```
❌ "API와 UI가 일치한다"
✅ "API 응답의 첫 번째 항목 {필드명}과 DOM의 {선택자} 텍스트가 완전히 같다"
```

---

#### 축 2 — 완전성 (Completeness): 빠진 항목이 없는가

**핵심 질문**: 표시되어야 할 항목이 하나도 빠짐없이 모두 표시되는가?

TC 도출 기준:
- 서비스가 "N개"를 보장한다면 정확히 N개인지 카운트한다
- 목록의 첫 항목과 마지막 항목이 모두 렌더링되는지 확인한다
- 스크롤/페이지네이션 후에도 데이터가 중복·누락 없이 유지되는지 확인한다

기대 결과 작성 기준:
```
❌ "목록이 표시된다"
✅ "총 {N}개 행이 렌더링되고, 행 식별자에 중복값이 없다 (Set.size === N)"
```

---

#### 축 3 — 정확성 (Accuracy): 계산·정렬·매핑이 올바른가

**핵심 질문**: 서비스가 자체적으로 계산·가공한 값이 정확한가?

TC 도출 기준:
- Q5에서 파악한 금전·법적 데이터의 계산식을 TC에 명시하고 검증한다
- 정렬 기준이 있다면 실제 순서가 기준과 일치하는지 확인한다 (상위 N개 비교)
- 연관 데이터 매핑 (상품 A의 이미지가 상품 A의 것인지 등)을 확인한다

기대 결과 작성 기준:
```
❌ "가격이 올바르다"
✅ "최종 판매가 = floor(원가 × (1 - 할인율))이고, API의 finalPrice와 일치한다"
```

---

#### 축 4 — 신뢰 신호 (Trust Signals): 사용자가 상황을 파악할 수 있는가

**핵심 질문**: 날짜·시간·상태·배지·수량 등 사용자 판단에 필요한 정보가 명확히 표시되는가?

TC 도출 기준:
- Q3, Q4에서 파악한 상태 표시·날짜 기준을 TC로 만든다
- 상태 배지(신규·품절·인기·할인 등)가 API 필드값과 일치하는지 확인한다
- 데이터 기준 날짜/시간이 화면에 표시되고 유효한 형식인지 확인한다
- 한글·특수문자 인코딩 깨짐(`\uFFFD`)이 없는지 확인한다

---

#### 축 5 — 안정성 (Stability): 어디서나 같은 결과인가

**핵심 질문**: 브라우저·기기·새로고침·스크롤 후에도 동일한 결과가 나오는가?

TC 도출 기준:
- Desktop(≥1280px) / Tablet(768px) / Mobile(375px) 각각에서 핵심 데이터 표시 확인
- Chrome / Edge / Safari 각각에서 핵심 데이터 일관성 확인
- 새로고침 후 데이터가 비정상적으로 변경되지 않는지 확인
- 스크롤 후 데이터가 재배열·중복되지 않는지 확인

---

#### 축 6 — 복원력 (Resilience): 오류 시 사용자를 안내하는가

**핵심 질문**: 외부 의존성이 실패했을 때 사용자가 길을 잃지 않는가?

TC 도출 기준:
- Q6에서 파악한 외부 의존성별로 실패 시나리오 TC를 만든다
- API 5xx → 흰 화면이 아닌 안내 메시지 또는 재시도 버튼
- 빈 데이터 → Empty State 안내 문구 (목록이 없을 때 무엇을 보여주는가)
- null/누락 필드 → JS 에러 없음, 레이아웃 붕괴 없음
- 이미지 CDN 실패 → Broken Image 0개 (`naturalWidth > 0`)
- 오프라인 → 안내 메시지 표시
- JS 콘솔 에러 → 정상 이용 흐름에서 `console.error` 0건

---

#### 축 7 — 성능 (Performance): 기다리지 않아도 될 만큼 빠른가

**핵심 질문**: 사용자가 서비스를 기다리다 포기하지 않을 만큼 빠른가?

TC 도출 기준:
- API 응답 시간: SLA 수치를 assert에 직접 명시 (`toBeLessThan(1_000)` 등)
- 페이지 로드: LCP ≤ 2,500ms (Core Web Vitals Good 기준)
- 연속 호출: N회 반복 시 편차가 허용 범위 내
- 지연 삽입 후에도 정상 완료되는지 확인 (Robust 겸용)

기대 결과 작성 기준:
```
❌ "응답이 빠르다"
✅ "elapsed < 3_000ms (toBeLessThan(3_000)); LCP < 2_500ms"
```

---

#### 축 8 — 접근성 (Accessibility): 누구나 사용할 수 있는가

**핵심 질문**: 키보드만으로도, 스크린 리더로도 서비스를 이용할 수 있는가?

TC 도출 기준:
- 키보드 Tab 탐색: 포커스가 주요 인터랙티브 요소(링크·버튼)를 순회
- 포커스 시각화: `outline: none` 단독 적용 없음 또는 대체 스타일 존재
- 이미지 alt: decorative가 아닌 이미지에 의미 있는 alt 텍스트 존재
- (권장) `@axe-core/playwright` 자동 검사

---

### Phase 3 — 검증 및 마무리

#### 3-1. 추적성 검토

추적성 매트릭스에서 계획서의 **모든 리스크·Exit Criteria**가 TC에 연결되어 있는지 확인한다.
연결 안 된 항목이 있으면 TC를 추가하거나 "N/A: {사유}"를 기재한다.

#### 3-2. 기대 결과 품질 검토

| 금지 표현 | 대체 기준 |
|-----------|----------|
| "일치한다" | API 응답의 특정 필드명과 DOM 선택자 값을 명시하고 비교 |
| "정상 표시된다" | 렌더링된 항목 수, naturalWidth 조건 등 수치로 명시 |
| "오류가 없다" | `console.error` 건수 = 0, `<body>` visible 유지 등 |
| "SLA를 만족한다" | `toBeLessThan(N)` 형태로 수치 직접 기재 |
| "이상 없다" | DOM 조건 또는 수치 조건으로 대체 |

#### 3-3. 자동화 매핑 기입

**비워두지 않는다.** Step 3 구현 전이라도 예상 파일명·메서드명·assert 예시를 기입한다.

#### 3-4. 우선순위 분류

| 우선순위 | 기준 |
|----------|------|
| **Critical** | CI 릴리즈 게이트. Smoke + Q1 핵심 데이터 정확성 + SLA |
| **High** | 축 1~6 핵심 TC. Sprint 내 자동화 완료 |
| **Medium** | 축 7~8, 반응형, 크로스 브라우저 |
